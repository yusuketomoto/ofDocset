43 /* The mesh structure is similar in spirit, notation, and operations
44 * to the "quad-edge" structure (see L. Guibas and J. Stolfi, Primitives
45 * for the manipulation of general subdivisions and the computation of
46 * Voronoi diagrams, ACM Transactions on Graphics, 4(2):74-123, April 1985).
47 * For a simplified description, see the course notes for CS348a,
48 * "Mathematical Foundations of Computer Graphics", available at the
49 * Stanford bookstore (and taught during the fall quarter).
50 * The implementation also borrows a tiny subset of the graph-based approach
51 * use in Mantyla's Geometric Work Bench (see M. Mantyla, An Introduction
52 * to Sold Modeling, Computer Science Press, Rockville, Maryland, 1988).
53 *
54 * The fundamental data structure is the "half-edge". Two half-edges
55 * go together to make an edge, but they point in opposite directions.
56 * Each half-edge has a pointer to its mate (the "symmetric" half-edge Sym),
57 * its origin vertex (Org), the face on its left side (Lface), and the
58 * adjacent half-edges in the CCW direction around the origin vertex
59 * (Onext) and around the left face (Lnext). There is also a "next"
60 * pointer for the global edge list (see below).
61 *
62 * The notation used for mesh navigation:
63 * Sym = the mate of a half-edge (same edge, but opposite direction)
64 * Onext = edge CCW around origin vertex (keep same origin)
65 * Dnext = edge CCW around destination vertex (keep same dest)
66 * Lnext = edge CCW around left face (dest becomes new origin)
67 * Rnext = edge CCW around right face (origin becomes new dest)
68 *
69 * "prev" means to substitute CW for CCW in the definitions above.
70 *
71 * The mesh keeps global lists of all vertices, faces, and edges,
72 * stored as doubly-linked circular lists with a dummy header node.
73 * The mesh stores pointers to these dummy headers (vHead, fHead, eHead).
74 *
75 * The circular edge list is special; since half-edges always occur
76 * in pairs (e and e->Sym), each half-edge stores a pointer in only
77 * one direction. Starting at eHead and following the e->next pointers
78 * will visit each *edge* once (ie. e or e->Sym, but not both).
79 * e->Sym stores a pointer in the opposite direction, thus it is
80 * always true that e->Sym->next->Sym->next == e.
81 *
82 * Each vertex has a pointer to next and previous vertices in the
83 * circular list, and a pointer to a half-edge with this vertex as
84 * the origin (NULL if this is the dummy header). There is also a
85 * field "data" for client data.
86 *
87 * Each face has a pointer to the next and previous faces in the
88 * circular list, and a pointer to a half-edge with this face as
89 * the left face (NULL if this is the dummy header). There is also
90 * a field "data" for client data.
91 *
92 * Note that what we call a "face" is really a loop; faces may consist
93 * of more than one loop (ie. not simply connected), but there is no
94 * record of this in the data structure. The mesh may consist of
95 * several disconnected regions, so it may not be possible to visit
96 * the entire mesh by starting at a half-edge and traversing the edge
97 * structure.
98 *
99 * The mesh does NOT support isolated vertices; a vertex is deleted along
100 * with its last edge. Similarly when two faces are merged, one of the
101 * faces is deleted (see tessMeshDelete below). For mesh operations,
102 * all face (loop) and vertex pointers must not be NULL. However, once
103 * mesh manipulation is finished, TESSmeshZapFace can be used to delete
104 * faces of the mesh, one at a time. All external faces can be "zapped"
105 * before the mesh is returned to the client; then a NULL face indicates
106 * a region which is not part of the output polygon.
107 */

170 /* The mesh operations below have three motivations: completeness,
171 * convenience, and efficiency. The basic mesh operations are MakeEdge,
172 * Splice, and Delete. All the other edge operations can be implemented
173 * in terms of these. The other operations are provided for convenience
174 * and/or efficiency.
175 *
176 * When a face is split or a vertex is added, they are inserted into the
177 * global list *before* the existing vertex or face (ie. e->Org or e->Lface).
178 * This makes it easier to process all vertices or faces in the global lists
179 * without worrying about processing the same data twice. As a convenience,
180 * when a face is split, the "inside" flag is copied from the old face.
181 * Other internal data (v->data, v->activeRegion, f->data, f->marked,
182 * f->trail, e->winding) is set to zero.
183 *
184 * ********************** Basic Edge Operations **************************
185 *
186 * tessMeshMakeEdge( mesh ) creates one edge, two vertices, and a loop.
187 * The loop (face) consists of the two new half-edges.
188 *
189 * tessMeshSplice( eOrg, eDst ) is the basic operation for changing the
190 * mesh connectivity and topology. It changes the mesh so that
191 * eOrg->Onext <- OLD( eDst->Onext )
192 * eDst->Onext <- OLD( eOrg->Onext )
193 * where OLD(...) means the value before the meshSplice operation.
194 *
195 * This can have two effects on the vertex structure:
196 * - if eOrg->Org != eDst->Org, the two vertices are merged together
197 * - if eOrg->Org == eDst->Org, the origin is split into two vertices
198 * In both cases, eDst->Org is changed and eOrg->Org is untouched.
199 *
200 * Similarly (and independently) for the face structure,
201 * - if eOrg->Lface == eDst->Lface, one loop is split into two
202 * - if eOrg->Lface != eDst->Lface, two distinct loops are joined into one
203 * In both cases, eDst->Lface is changed and eOrg->Lface is unaffected.
204 *
205 * tessMeshDelete( eDel ) removes the edge eDel. There are several cases:
206 * if (eDel->Lface != eDel->Rface), we join two loops into one; the loop
207 * eDel->Lface is deleted. Otherwise, we are splitting one loop into two;
208 * the newly created loop will contain eDel->Dst. If the deletion of eDel
209 * would create isolated vertices, those are deleted as well.
210 *
211 * ********************** Other Edge Operations **************************
212 *
213 * tessMeshAddEdgeVertex( eOrg ) creates a new edge eNew such that
214 * eNew == eOrg->Lnext, and eNew->Dst is a newly created vertex.
215 * eOrg and eNew will have the same left face.
216 *
217 * tessMeshSplitEdge( eOrg ) splits eOrg into two edges eOrg and eNew,
218 * such that eNew == eOrg->Lnext. The new vertex is eOrg->Dst == eNew->Org.
219 * eOrg and eNew will have the same left face.
220 *
221 * tessMeshConnect( eOrg, eDst ) creates a new edge from eOrg->Dst
222 * to eDst->Org, and returns the corresponding half-edge eNew.
223 * If eOrg->Lface == eDst->Lface, this splits one loop into two,
224 * and the newly created loop is eNew->Lface. Otherwise, two disjoint
225 * loops are merged into one, and the loop eDst->Lface is destroyed.
226 *
227 * ************************ Other Operations *****************************
228 *
229 * tessMeshNewMesh() creates a new mesh with no edges, no vertices,
230 * and no loops (what we usually call a "face").
231 *
232 * tessMeshUnion( mesh1, mesh2 ) forms the union of all structures in
233 * both meshes, and returns the new mesh (the old meshes are destroyed).
234 *
235 * tessMeshDeleteMesh( mesh ) will free all storage for any valid mesh.
236 *
237 * tessMeshZapFace( fZap ) destroys a face and removes it from the
238 * global face list. All edges of fZap will have a NULL pointer as their
239 * left face. Any edges which also have a NULL pointer as their right face
240 * are deleted entirely (along with any isolated vertices this produces).
241 * An entire mesh can be deleted by zapping its faces, one at a time,
242 * in any order. Zapped faces cannot be used in further mesh operations!
243 *
244 * tessMeshCheckMesh( mesh ) checks a mesh for self-consistency.
245 */
